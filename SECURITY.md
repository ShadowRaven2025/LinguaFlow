## 🚨 Сообщение об уязвимостях

Безопасность наших пользователей - наш главный приоритет. Если вы обнаружили уязвимость в LinguaFlow, пожалуйста, сообщите нам об этом ответственно.

### Как сообщить

**НЕ создавайте публичные GitHub Issues для уязвимостей безопасности.**

Вместо этого:

1. **Email**: Отправьте подробный отчет на security@linguaflow.com
2. **Encrypted Email**: Используйте наш PGP ключ для шифрования (доступен по запросу)
3. **GitHub Security Advisory**: Используйте функцию "Report a vulnerability" в разделе Security

### Информация для включения

Пожалуйста, включите следующую информацию в ваш отчет:

- **Тип уязвимости** (например, XSS, SQL injection, CSRF)
- **Полные пути к исходным файлам**, связанным с проявлением уязвимости
- **Расположение** затронутого исходного кода (тег/ветка/коммит или прямой URL)
- **Любая специальная конфигурация**, необходимая для воспроизведения проблемы
- **Пошаговые инструкции** для воспроизведения проблемы
- **Proof-of-concept или эксплойт код** (если возможно)
- **Влияние уязвимости**, включая то, как злоумышленник может использовать проблему

### Что ожидать

- **Подтверждение получения** в течение 24 часов
- **Первоначальная оценка** в течение 72 часов
- **Регулярные обновления** о прогрессе исправления
- **Публичное раскрытие** после исправления (с вашего согласия)

## 🔒 Меры безопасности

### Архитектура безопасности

LinguaFlow реализует несколько уровней безопасности:

#### 1. Аутентификация и авторизация
- **Supabase Auth** для управления пользователями
- **JWT токены** для сессий
- **Row Level Security (RLS)** в базе данных
- **Защищенные API маршруты**

#### 2. Защита данных
- **Шифрование в покое** через Supabase
- **HTTPS** для всех соединений
- **Валидация данных** на клиенте и сервере
- **Санитизация пользовательского ввода**

#### 3. Защита от атак
- **CSRF защита** через SameSite cookies
- **XSS защита** через Content Security Policy
- **SQL Injection защита** через параметризованные запросы
- **Rate limiting** для API эндпоинтов

#### 4. Мониторинг и логирование
- **Аудит действий пользователей**
- **Мониторинг подозрительной активности**
- **Логирование ошибок безопасности**

### Конфигурация безопасности

#### Environment Variables
```bash
# Никогда не коммитьте эти переменные
SUPABASE_SERVICE_ROLE_KEY=your_secret_key
DATABASE_PASSWORD=your_db_password

# Используйте только в .env.local
NEXT_PUBLIC_SUPABASE_URL=your_public_url
NEXT_PUBLIC_SUPABASE_PUBLISHABLE_DEFAULT_KEY=your_public_key
```

#### Headers безопасности
```javascript
// next.config.js
const securityHeaders = [
  {
    key: 'X-DNS-Prefetch-Control',
    value: 'on'
  },
  {
    key: 'Strict-Transport-Security',
    value: 'max-age=63072000; includeSubDomains; preload'
  },
  {
    key: 'X-XSS-Protection',
    value: '1; mode=block'
  },
  {
    key: 'X-Frame-Options',
    value: 'SAMEORIGIN'
  },
  {
    key: 'X-Content-Type-Options',
    value: 'nosniff'
  },
  {
    key: 'Referrer-Policy',
    value: 'origin-when-cross-origin'
  }
]
```

## 🔍 Аудит безопасности

### Регулярные проверки

Мы проводим регулярные аудиты безопасности:

- **Автоматическое сканирование зависимостей** через GitHub Dependabot
- **Статический анализ кода** через ESLint security правила
- **Проверка уязвимостей** через npm audit
- **Мануальный код-ревью** с фокусом на безопасность

### Инструменты безопасности

```bash
# Проверка уязвимостей в зависимостях
bun audit

# Статический анализ безопасности
bun run lint:security

# Проверка секретов в коде
git-secrets --scan
```

### План реагирования

В случае инцидента безопасности:

1. **Немедленная изоляция** - остановить распространение
2. **Оценка ущерба** - определить масштаб проблемы

5. **Восстановление** - восстановить нормальную работу
6. **Анализ** - провести post-mortem анализ
